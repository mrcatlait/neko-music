# Platform ships as a library and serves its own web client

In the final version, Neko Music is distributed as a single installable package (`@neko/platform`, with a thin `@neko/strategy-sdk` for Strategy authors) rather than as a standalone server app. An operator's installation is a small scaffold project — generated by a Neko CLI — whose entry point calls `bootstrap(config)`. The platform bundles the single-page web client and serves it itself, so an installation is one self-contained deployable: back end and web client together, no separate SPA host.

This replaces today's two-app shape (`apps/server` + separately-hosted `apps/web`). We chose it to make hosting trivial (run one thing) and to keep the web client and API versioned in lockstep. The model follows Vendure's `VendureConfig` + `bootstrap()` + `@vendure/create` precedent.

**Considered options:** keep server and SPA as two separately-hosted artifacts (rejected — operator must host two things, and the boundary invites version drift); SSR metaframework serving the UI (rejected — see the private/no-SEO posture; SSR adds hosting complexity and untrusted-render surface for no benefit here).

**Consequences:** the repo reshapes around a published platform package + thin scaffolded app; the web client moves inside the platform package; the CLI becomes a maintained surface.
